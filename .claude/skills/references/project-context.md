# Project Context

## コードベース調査ガイド

### モジュール構成の把握方法

`go.mod` を読み、ルートパッケージ `github.com/bright-room/skill-test-sandbox` の構成を確認する。サブパッケージはない。

### 既存パターンの調査手順

- 関数の実装パターン: `greet.go` を参照（引数が空文字の場合のデフォルト値パターン）
- エラーハンドリングパターン: `math.go` を参照（`(T, error)` シグネチャ）

### テスト構成の確認方法

- テストは `*_test.go` ファイルに配置（Go 標準）
- `go test ./...` で全テスト実行
- テーブルドリブンテスト: `greet_test.go` を参照

## 実装ガイド

### ビルド・フォーマットコマンド

| コマンド | 用途 | タイミング |
|---------|------|-----------|
| `go build ./...` | ビルド確認 | 実装後 |
| `go test ./...` | テスト実行 | 実装後 |
| `gofmt -w .` | フォーマット | コミット前（必須） |

### 言語固有の実装規約

- GoDoc コメントは英語で記述する
- エクスポートされる関数には必ず GoDoc コメントをつける
- エラーは `errors.New()` または `fmt.Errorf()` で生成する

### テスト配置ルール

- テストファイルは対象ファイルと同じディレクトリに `<name>_test.go` として配置する
- テスト関数名は `Test<FunctionName>` とする
- テーブルドリブンテスト（`[]struct` + `t.Run`）を使用する

### CI に委ねてよい項目

なし（全テストをローカルで実行可能）

## レビューガイド

### ファイルパス → カテゴリマッピング

| 変更ファイルのパスパターン | 選択されるカテゴリ |
|--------------------------|-------------------|
| `*.go`（`*_test.go` 除く） | code |
| `*_test.go` | test |
| `go.mod` | build |
| `CLAUDE.md`, `README.md` | docs |

### カテゴリ別レビュー観点

#### code
- 既存関数と同じパターンに従っているか
- エラーハンドリングが適切か（ゼロ値チェック等）
- GoDoc コメントが英語で記述されているか

#### test
- テーブルドリブンテストを使用しているか
- 正常系・異常系の両方をカバーしているか
- テスト関数名が `Test<FunctionName>` に従っているか

#### build
- 不要な依存が追加されていないか

#### docs
- CLAUDE.md の内容が実態と一致しているか
